# Требования

## Цель продукта

PEP - образовательная платформа для практического обучения информационной безопасности.
Студент изучает тему из Академии web-безопасности, загружает архив уязвимого стенда или готовый
Docker image, доказывает уязвимость в white box отчете, запускает системные стенды и тестирует чужие
приложения в black box формате.

## Языковые требования

- Весь пользовательский интерфейс должен быть на русском языке.
- Все учебные материалы, включая Docker-курс, теорию по уязвимостям, примеры и шаблоны отчетов,
  должны быть на русском языке.
- Проектная документация и README должны быть на русском языке.
- Английские технические термины допускаются, если они являются общепринятыми: Docker, image,
  container, Kubernetes, namespace, ingress, white box, black box, payload, PoC.

## Целевая стратегия

Система проектируется как production-ready платформа, но дипломная демонстрация должна запускаться
локально на компьютере через `kind` и Docker.

## Роли

### Студент

- изучает материалы курса и примеры уязвимого кода;
- проходит вводный курс по Docker;
- создает уязвимое веб-приложение;
- загружает архив стенда или отправляет готовый Docker image;
- отправляет white box отчет;
- запускает официальные системные стенды на 4 часа;
- получает три чужих приложения для black box тестирования;
- пишет black box отчеты;
- получает оценку и комментарии куратора.

### Куратор

- проверяет white box и black box отчеты;
- подтверждает или отклоняет найденные уязвимости;
- выставляет баллы по чеклисту;
- возвращает отчеты на доработку;
- оставляет техническую обратную связь.

### Администратор

- управляет пользователями, группами, курсами и модулями;
- настраивает дедлайны и фазы модуля;
- запускает этап black box распределения;
- отслеживает состояние lab-контейнеров;
- просматривает аудит и экспортирует результаты.

## Границы MVP

MVP должен включать:

- авторизацию и role-based access control;
- один курс с вводным модулем по Docker;
- набор модулей Академии web-безопасности;
- сдачу приложения через архив с `Dockerfile`/`docker-compose.yml` или Docker image reference;
- техническую проверку: сборка/pull image, запуск контейнера, доступность порта, базовый health/status;
- white box отчет;
- запуск утвержденного image в локальном Kubernetes через `kind`;
- официальные системные стенды с запуском на 4 часа;
- проверку white box отчета куратором;
- распределение black box заданий;
- black box отчет;
- проверку и оценку куратором;
- базовую административную панель;
- базовый audit log;
- resource limits и network isolation для lab-контейнеров.

## Production scope

Production-ready версия добавляет:

- очередь задач для проверки и деплоя image;
- приватный container registry;
- S3-compatible storage для отчетов и вложений;
- строгие Kubernetes policy для недоверенных контейнеров;
- observability dashboards и alerting;
- backup и restore;
- rate limiting и security hardening;
- image scanning и dependency scanning;
- аудит, пригодный для разбора инцидентов.

## Функциональные требования

### Пользовательские сценарии

- Для студента, куратора и администратора должны быть описаны user stories и критерии приемки.
- Каждый ключевой сценарий MVP должен быть проверяемым через UI или API.
- User stories должны быть связаны с ролями, отчетами, validation jobs и lab runtime.

### Курсы и материалы

- Система должна поддерживать курсы, модули, уроки и темы уязвимостей.
- Уроки должны содержать Markdown-материалы и примеры уязвимого кода.
- Администратор должен публиковать и скрывать учебные материалы.

### Академия web-безопасности

- Первая полная программа должна покрывать практические web-уязвимости: SQL-инъекции, XSS, SSTI,
  SSRF, CSRF, CORS, XXE, бизнес-логику, BAC и IDOR.
- Для каждой темы должны быть: теория, типы уязвимости, методики эксплуатации, пример уязвимого кода,
  практическое задание, критерии white box отчета и критерии black box отчета.

### Docker-курс

- Вводный курс по Docker должен быть обязательным перед практическими лабораториями.
- Курс должен объяснять Dockerfile, сборку image, запуск контейнера, порты, environment variables,
  volumes, Docker Compose, registry и базовую безопасность контейнеров.

### Сдача приложения

- Основной формат сдачи: архив проекта с `Dockerfile` или `docker-compose.yml`.
- Fallback-формат: Docker image reference.
- Студент должен указать порт приложения, опциональный health endpoint и README.
- Платформа должна собрать archive submission в local registry.
- White box отчет прикладывается к сдаче.
- Система должна отслеживать статусы технической проверки и проверки куратором.

### Техническая проверка image

- Worker должен выполнить pull image.
- Worker должен запустить контейнер с лимитами ресурсов.
- Worker должен проверить, что приложение стартует за заданный timeout.
- Worker должен проверить доступность указанного порта.
- Worker должен проверить health endpoint, если он указан.
- Worker не должен автоматически доказывать наличие уязвимости.

### Lab runtime

- Утвержденные сдачи должны запускаться как изолированные lab instances в локальном `kind`-кластере.
- Каждый lab instance должен иметь CPU/RAM limits.
- Каждый lab instance должен иметь route через ingress или port-forward для локальной демонстрации.
- Lab instances должны удаляться после завершения модуля или истечения TTL.

### Black box задания

- Администратор запускает распределение после появления достаточного числа утвержденных работ.
- Каждый студент получает до трех чужих приложений.
- Студент никогда не получает собственное приложение.
- Результат распределения фиксируется после старта black box фазы.

### Отчеты и проверки

- Отчеты должны поддерживать Markdown и вложения.
- Куратор проверяет отчеты по чеклисту и выставляет баллы.
- Статусы отчета: черновик, отправлен, на проверке, принят, отклонен, нужна доработка.
- Студент видит комментарии куратора после проверки.
- Критерии оценивания Docker image, white box отчета и black box отчета должны быть заранее
  опубликованы для студентов и кураторов.

### API contract

- Backend должен иметь согласованный REST API contract для frontend и worker.
- MVP API должен покрывать auth, courses, lessons, submissions, validation jobs, labs, reports,
  reviews, black box assignments и audit.
- Ошибки API должны возвращаться в едином JSON-формате.

## Нефункциональные требования

- Backend API должен проверять права доступа на уровне каждого объекта.
- Долгие операции должны выполняться асинхронно.
- Backend request handlers не должны напрямую выполнять тяжелые Docker/Kubernetes операции.
- Пользовательские контейнеры считаются недоверенным кодом.
- Важные действия должны попадать в audit log.
- Система должна отдавать health и metrics endpoints.
- Все изменения persistent-схемы должны версионироваться миграциями.
