# Учебная программа OWASP Top 10

## Цель

Сформировать базовое понимание наиболее распространенных web-уязвимостей и дать практику в двух
форматах: white box создание уязвимого приложения и black box тестирование чужого приложения.

## Формат каждого модуля

Каждый модуль содержит:

- теорию на русском языке;
- объяснение типовых причин уязвимости;
- пример уязвимого кода;
- пример исправленного кода;
- white box задание;
- шаблон white box отчета;
- black box чеклист;
- критерии оценки.

## Темы курса

### A01. Broken Access Control

Студент изучает ошибки контроля доступа: IDOR, отсутствие проверки владельца объекта, обход ролей.

Практика:

- создать приложение, где пользователь может получить чужой объект по ID;
- доказать проблему в white box отчете;
- в black box режиме найти объект, к которому не должно быть доступа.

### A02. Cryptographic Failures

Студент изучает ошибки хранения и передачи чувствительных данных.

Практика:

- показать небезопасное хранение секрета или слабое хеширование;
- описать impact;
- предложить корректное исправление.

### A03. Injection

Основная демонстрационная тема для MVP. Включает SQL Injection, Command Injection и базовые идеи
инъекций в query/template/context.

Практика:

- создать приложение с SQL Injection;
- опубликовать Docker image;
- приложить payload и доказательство эксплуатации;
- найти SQL Injection в чужом приложении.

### A04. Insecure Design

Студент изучает ошибки проектирования, которые нельзя исправить одной проверкой input validation.

Практика:

- реализовать бизнес-логику с обходом лимита или workflow;
- описать, почему проблема находится на уровне дизайна;
- предложить безопасный процесс.

### A05. Security Misconfiguration

Студент изучает опасные настройки, debug endpoints, default credentials, лишние headers и открытые
административные интерфейсы.

Практика:

- создать приложение с включенным debug endpoint или default credentials;
- доказать риск;
- предложить hardening.

### A06. Vulnerable and Outdated Components

Студент изучает риск устаревших зависимостей.

Практика:

- использовать заведомо уязвимую библиотеку в учебном примере;
- показать CVE или публичное описание уязвимости;
- описать процесс обновления.

### A07. Identification and Authentication Failures

Студент изучает ошибки аутентификации: слабые пароли, predictable reset tokens, неправильные session
settings.

Практика:

- реализовать слабый reset token или небезопасную session-механику;
- доказать эксплуатацию;
- предложить исправление.

### A08. Software and Data Integrity Failures

Студент изучает доверие к неподписанным данным, небезопасные update-механизмы и insecure
deserialization на базовом уровне.

Практика:

- показать небезопасную обработку доверенных данных;
- описать сценарий подмены;
- предложить integrity check.

### A09. Security Logging and Monitoring Failures

Студент изучает, почему отсутствие логирования и мониторинга мешает обнаруживать атаки.

Практика:

- создать сценарий атаки, который не оставляет полезных событий;
- предложить набор логов и alert-условий.

### A10. Server-Side Request Forgery

Студент изучает SSRF и риски доступа к внутренним ресурсам через сервер.

Практика:

- реализовать учебный SSRF endpoint;
- ограничить scope атаки безопасной локальной целью;
- описать allowlist и сетевые ограничения.

## Текущий demo-seed

Для защиты диплома полностью демонстрируется модуль `A03. Injection`: Docker image submission,
technical validation, white box отчет, запуск lab, black box distribution и проверка отчета.

Остальные темы OWASP Top 10 также создаются в seed данных как активные учебные модули. Для каждой
темы есть базовый набор из трех уроков: теория и scope, white box lab guidance, black box чеклист.
Это позволяет открыть полный каталог OWASP Top 10 в интерфейсе и использовать один общий workflow
сдачи Docker image и отчетов для дальнейшего расширения практики.

## Подробный MVP-модуль A03. Injection

### Цель модуля

Научить студента понимать, создавать, эксплуатировать и описывать injection-уязвимости на примере
SQL Injection и Command Injection. Основной демонстрационный сценарий для защиты - SQL Injection.

### Урок 1. Что такое Injection

Теория:

- доверие к пользовательскому вводу;
- смешивание данных и команд;
- почему escaping и parameterized queries отличаются;
- типовые последствия: обход авторизации, чтение данных, изменение данных, выполнение команд.

Практика:

- разобрать псевдокод уязвимого запроса;
- определить, где пользовательский ввод попадает в командный контекст.

### Урок 2. SQL Injection

Теория:

- строковая конкатенация SQL;
- boolean-based SQL Injection;
- error-based признаки;
- login bypass;
- parameterized queries как исправление.

Пример уязвимого кода:

```java
String sql = "SELECT * FROM users WHERE email = '" + email + "' AND password = '" + password + "'";
```

Пример исправления:

```java
PreparedStatement statement = connection.prepareStatement(
    "SELECT * FROM users WHERE email = ? AND password = ?"
);
statement.setString(1, email);
statement.setString(2, password);
```

White box задание:

- создать web-приложение с формой входа или поиска;
- намеренно реализовать SQL Injection;
- подготовить Docker image;
- описать уязвимый участок кода;
- показать payload и результат эксплуатации.

Минимальный payload для демонстрации:

```text
' OR '1'='1
```

### Урок 3. Command Injection

Теория:

- передача пользовательского ввода в shell-команду;
- command separators;
- allowlist vs blacklist;
- безопасные API без shell.

Практика:

- разобрать пример endpoint, который выполняет `ping`;
- показать, почему конкатенация команды опасна;
- предложить allowlist допустимых значений.

### Урок 4. Исправление Injection

Теория:

- parameterized queries;
- ORM limitations;
- input validation;
- output encoding не исправляет SQL Injection;
- принцип наименьших привилегий для database user.

Практика:

- переписать уязвимый SQL-запрос на parameterized query;
- ограничить database permissions;
- описать, почему исправление работает.

### Урок 5. White box отчет

Студент должен описать:

- файл и участок уязвимого кода;
- путь пользовательского ввода до SQL/command контекста;
- payload;
- шаги воспроизведения;
- доказательство успешной эксплуатации;
- impact;
- исправление.

### Урок 6. Black box чеклист

При тестировании чужого приложения студент проверяет:

- формы входа;
- поиск;
- фильтры и сортировки;
- numeric и string parameters;
- ошибки базы данных;
- различия в ответах на boolean payloads;
- признаки command execution в endpoint, которые принимают host, domain, filename или path.

### Acceptance criteria Docker image

- image доступен по reference;
- container стартует без ручных действий;
- приложение слушает указанный port;
- приложение отвечает на smoke check;
- image не требует `--privileged`;
- image не требует `--network host`;
- README описывает тестовые учетные данные, если они нужны.

### Acceptance criteria white box отчета

- уязвимость соответствует Injection;
- payload воспроизводим;
- есть доказательство эксплуатации;
- указан уязвимый код;
- объяснена причина уязвимости;
- предложено корректное исправление.

### Acceptance criteria black box отчета

- цель соответствует назначенному lab;
- шаги воспроизведения понятны;
- payload указан полностью;
- evidence достаточно для проверки;
- severity обоснована;
- рекомендации по исправлению применимы.

## Шаблон white box отчета

- Название уязвимости.
- Где находится уязвимый код.
- Почему код уязвим.
- Payload или сценарий эксплуатации.
- Доказательство успешной эксплуатации.
- Impact.
- Как исправить.

## Шаблон black box отчета

- Цель тестирования.
- Найденная уязвимость.
- Severity.
- Шаги воспроизведения.
- Payload.
- Доказательства.
- Impact.
- Рекомендации по исправлению.
- Ограничения тестирования.
