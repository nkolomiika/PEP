# Критерии оценивания

## Цель

Документ фиксирует прозрачную систему оценивания для Docker-части, white box отчета и black box
отчета. Он нужен, чтобы студент понимал ожидания, а куратор проверял работы единообразно.

## Docker image и technical validation

Максимум: 100 баллов.

| Критерий | Баллы |
| --- | --- |
| Image reference доступен для pull | 20 |
| Container стартует без ручных действий | 20 |
| Приложение слушает указанный port | 20 |
| Health endpoint отвечает, если указан | 10 |
| Image не требует privileged mode и host network | 20 |
| README описывает запуск и тестовые данные | 10 |

Минимум для допуска к проверке куратора: technical validation должен завершиться статусом `PASSED`.

## White box отчет

Максимум: 100 баллов.

| Критерий | Баллы |
| --- | --- |
| Уязвимость соответствует теме модуля | 15 |
| Указан конкретный уязвимый код | 15 |
| Объяснена причина уязвимости | 15 |
| Payload воспроизводим | 15 |
| Есть доказательство эксплуатации | 15 |
| Impact описан реалистично | 10 |
| Предложено корректное исправление | 10 |
| Отчет структурирован и понятен | 5 |

Решения куратора:

- `APPROVED` - уязвимость воспроизводится, отчет достаточен;
- `NEEDS_REVISION` - есть исправимые недочеты;
- `REJECTED` - уязвимость не соответствует теме или не воспроизводится.

## Black box отчет

Максимум: 100 баллов.

| Критерий | Баллы |
| --- | --- |
| Цель соответствует назначенному lab | 10 |
| Уязвимость валидна | 20 |
| Шаги воспроизведения полные | 15 |
| Payload указан корректно | 15 |
| Evidence достаточно для проверки | 15 |
| Severity обоснована | 10 |
| Рекомендации по исправлению применимы | 10 |
| Отчет профессионально оформлен | 5 |

## Итоговая оценка модуля

Рекомендуемая формула:

```text
Итог = Docker допуск + WhiteBox * 0.45 + BlackBox * 0.55
```

Docker-часть является допуском. Если technical validation не пройден, модуль не может быть закрыт.

## Severity

Для MVP достаточно четырех уровней:

- `Low` - ограниченное влияние, эксплуатация затруднена;
- `Medium` - заметное влияние на данные или workflow;
- `High` - доступ к чужим данным, обход авторизации или изменение данных;
- `Critical` - выполнение команд, полный обход контроля доступа или компрометация системы.

## Требования к evidence

Evidence может включать:

- скриншот;
- HTTP request/response;
- payload;
- фрагмент лога;
- описание наблюдаемого результата.

Evidence не должен содержать реальные персональные данные или секреты.
