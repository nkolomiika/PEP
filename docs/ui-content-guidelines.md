# UI content guidelines

## Общие правила

- Весь пользовательский интерфейс пишется на русском языке.
- Технические термины можно оставлять на английском, если они являются стандартом: `Docker image`,
  `black box`, `white box`, `payload`, `health check`, `port-forward`.
- Сообщения должны объяснять действие пользователя, а не внутреннюю ошибку платформы.
- Статусы должны совпадать с `docs/status-and-errors.md`.

## Labels и кнопки

- Использовать глаголы действия: `Отправить отчет`, `Повторить проверку`, `Открыть lab`.
- Избегать неопределенных кнопок: `OK`, `Submit`, `Go`.
- Для опасных действий использовать подтверждение: `Остановить lab`, `Перезапустить проверку`.

## Ошибки

Ошибка должна содержать:

- что произошло;
- как исправить;
- ссылку на детали, если они есть.

Пример:

```text
Техническая проверка не пройдена. Проверьте, что image доступен в registry и приложение отвечает на /health.
```

## Review comments

Комментарии куратора должны быть конкретными:

- что не принято;
- где это видно в отчете;
- какое evidence нужно добавить;
- какой дедлайн исправления.

## Admin actions

Административные экраны должны показывать impact действия:

- сколько students затронуто;
- какие labs будут перезапущены;
- можно ли отменить действие.

## Checklist

- Нет UI-facing английских фраз без причины.
- Все статусы имеют русские labels.
- Все формы имеют понятные ошибки.
- Security термины используются одинаково во всех страницах.
