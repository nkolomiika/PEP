# User stories

## Цель

Документ фиксирует основные пользовательские сценарии платформы. Он нужен для разработки MVP,
проверки полноты требований и защиты диплома.

## Студент

### Прохождение Docker-курса

Как студент, я хочу пройти вводный курс по Docker, чтобы подготовить приложение к сдаче в виде
архива стенда или Docker image.

Критерии приемки:

- студент видит список уроков Docker-курса;
- студент открывает уроки на русском языке;
- студент видит практическое задание;
- после изучения курса студент понимает, как собрать и запустить стенд.

### Изучение Академии web-безопасности

Как студент, я хочу изучить тему из Академии web-безопасности, чтобы понять причину уязвимости и способы ее
эксплуатации.

Критерии приемки:

- студент видит теорию на русском языке;
- студент видит типы уязвимости и методики эксплуатации;
- студент видит пример уязвимого кода;
- студент видит пример исправления;
- студент видит требования к white box отчету.

### Сдача уязвимого приложения

Как студент, я хочу отправить архив стенда или Docker image reference, port и white box отчет, чтобы куратор мог
проверить мою работу.

Критерии приемки:

- форма принимает archive upload;
- форма принимает fallback image reference;
- форма принимает application port;
- форма принимает optional health path;
- форма принимает Markdown-отчет;
- после отправки создается validation job;
- студент видит статус технической проверки.

### Запуск официального системного стенда

Как студент, я хочу запустить подготовленный преподавателем стенд, чтобы потренироваться на
контролируемой задаче.

Критерии приемки:

- студент видит каталог задач по категориям;
- студент запускает стенд на 4 часа;
- студент получает поддомен стенда;
- по истечении TTL стенд считается истекшим.

### Black box тестирование

Как студент, я хочу получить чужие lab-приложения, чтобы потренироваться находить уязвимости без
доступа к исходному коду.

Критерии приемки:

- студент видит список назначенных целей;
- студент не видит автора приложения;
- студент не получает собственное приложение;
- студент открывает lab через URL или port-forward;
- студент отправляет black box отчет.

## Куратор

### Проверка white box отчета

Как куратор, я хочу видеть очередь white box отчетов, чтобы подтверждать корректность заявленных
уязвимостей.

Критерии приемки:

- куратор видит список отчетов на проверку;
- куратор открывает отчет и сведения о technical validation;
- куратор выставляет решение: принять, отклонить, вернуть на доработку;
- куратор оставляет комментарий;
- решение сохраняется в audit log.

### Проверка black box отчета

Как куратор, я хочу проверять black box отчеты, чтобы оценивать качество найденных студентом
уязвимостей.

Критерии приемки:

- куратор видит цель тестирования;
- куратор видит payload, evidence и шаги воспроизведения;
- куратор выставляет баллы по rubric;
- студент видит итоговую обратную связь.

## Администратор

### Управление курсами и пользователями

Как администратор, я хочу создавать курсы, модули, страницы и пользователей, чтобы управлять учебным процессом.

Критерии приемки:

- администратор создает курс;
- администратор создает модуль;
- администратор редактирует Markdown-страницу;
- администратор создает и отключает пользователя.

### Управление системными задачами

Как администратор, я хочу публиковать и обновлять системные задачи, чтобы студенты запускали
официальные стенды без ручного внесения в БД.

Критерии приемки:

- администратор сохраняет/обновляет задачу;
- система публикует задачу в каталоге;
- задача появляется в разделе стендов;
- студент может запустить задачу.

### Запуск black box распределения

Как администратор, я хочу запустить распределение lab-приложений, чтобы студенты получили чужие
цели для тестирования.

Критерии приемки:

- распределение запускается только после появления approved submissions;
- система не назначает студенту собственный lab;
- система назначает до трех целей;
- результат распределения сохраняется;
- действие попадает в audit log.

### Контроль lab runtime

Как администратор, я хочу видеть состояние lab-приложений в `kind`, чтобы демонстрировать запуск
контейнеров и ограничения ресурсов.

Критерии приемки:

- администратор видит статус lab instance;
- администратор видит route или port-forward инструкцию;
- администратор видит resource limits;
- администратор может перезапустить или остановить lab.
