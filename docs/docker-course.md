# Вводный курс по Docker

## Цель

Курс готовит студента к публикации уязвимого приложения в виде готового Docker image, который
платформа сможет технически проверить и запустить в локальном Kubernetes-кластере через `kind`.

## Результаты обучения

После курса студент должен уметь:

- объяснить разницу между image и container;
- написать базовый Dockerfile;
- собрать image локально;
- запустить container локально;
- пробросить порт приложения;
- передать настройки через environment variables;
- читать логи и заходить внутрь container;
- использовать volumes для локальной разработки;
- запускать несколько сервисов через Docker Compose;
- опубликовать image в registry;
- подготовить image reference для сдачи на платформе;
- избегать опасных настроек: privileged mode, host network, hostPath.

## Структура курса

### Урок 1. Контейнеры и образы

Теория:

- зачем нужна контейнеризация;
- чем container отличается от image;
- что такое слои image;
- жизненный цикл container;
- что такое registry.

Практика:

- запустить `hello-world`;
- скачать и запустить nginx;
- посмотреть логи container.

Чеклист:

- студент объясняет image vs container;
- студент умеет запустить и остановить container.

### Урок 2. Основы Dockerfile

Теория:

- `FROM`;
- `WORKDIR`;
- `COPY`;
- `RUN`;
- `ENV`;
- `EXPOSE`;
- `CMD`;
- `ENTRYPOINT`.

Практика:

- создать простое web-приложение;
- написать Dockerfile;
- собрать image;
- запустить container.

Чеклист:

- image собирается без ручных действий;
- container стартует одной командой;
- приложение слушает документированный порт.

### Урок 3. Порты, переменные окружения и логи

Теория:

- port внутри container и port на host;
- environment variables;
- диагностика через logs и exec.

Практика:

- запустить приложение с `-p`;
- передать настройку через `-e`;
- посмотреть `docker logs`;
- использовать `docker exec`.

Чеклист:

- студент открывает приложение локально;
- студент настраивает приложение без изменения кода.

### Урок 4. Volumes и Docker Compose

Теория:

- bind mounts;
- named volumes;
- зависимости сервисов;
- сети Docker Compose.

Практика:

- создать `docker-compose.yml` с приложением и базой данных;
- сохранить состояние базы через volume;
- подключить приложение к базе по имени сервиса.

Чеклист:

- студент запускает multi-container приложение;
- студент понимает, почему `localhost` между containers не работает как на host.

### Урок 5. Публикация image

Теория:

- image tag;
- registry;
- `docker login`;
- `docker tag`;
- `docker push`;
- image reference.

Практика:

- собрать image с тегом;
- опубликовать image в registry;
- скачать image по reference;
- запустить скачанный image.

Чеклист:

- студент может получить корректный image reference;
- опубликованный image запускается на другой машине или в другом окружении.

### Урок 6. Безопасность и правила сдачи

Теория:

- почему нельзя требовать privileged mode;
- почему нельзя использовать host network;
- почему нельзя хранить секреты в image;
- зачем нужны CPU/RAM limits;
- что такое health endpoint.

Практика:

- добавить non-root user, если это возможно;
- проверить, что приложение стартует без интерактивного ввода;
- указать port и опциональный health endpoint;
- выполнить финальный smoke test.

Чеклист:

- приложение стартует без ручных действий;
- приложение слушает указанный port;
- image не требует privileged mode;
- image не требует host network;
- startup-инструкция описана в README.

## Итоговое практическое задание

Студент должен подготовить уязвимое web-приложение и сдать:

- Docker image reference;
- порт приложения;
- опциональный health endpoint;
- README с кратким описанием запуска;
- white box отчет по уязвимости.

## Критерии оценки Docker-части

- Корректный image reference: 20 баллов.
- Container успешно запускается: 25 баллов.
- Корректно указан port: 15 баллов.
- Приложение отвечает на smoke check: 20 баллов.
- Базовая безопасность container: 20 баллов.

Проходной балл: 70.

## Типичные ошибки

- использовать `localhost` для связи между containers;
- не указать port приложения;
- публиковать image, который зависит от локальных файлов;
- хранить секреты внутри image;
- требовать ручную инициализацию базы;
- требовать `--privileged`;
- требовать `--network host`;
- запускать приложение только от root без необходимости.
